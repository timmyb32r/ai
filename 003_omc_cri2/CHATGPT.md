# Протокол потоковой передачи

Для надежной трансляции живого радио с буферизацией и возможностью перемотки рекомендуются адаптивные HTTP-протоколы, такие как **HLS (HTTP Live Streaming)** или MPEG‑DASH. При HLS поток разбивается на небольшие сегменты (обычно по несколько секунд) и публикуется в плейлисте M3U8, который клиент периодически запрашивает. Это позволяет «склеивать» сегменты в непрерывный звук и плавно восстанавливать воспроизведение при смене качества или потере пакетов. Такой подход обеспечивает высокую стабильность и совместимость, особенно на мобильных устройствах: большинство Android- и iOS-плееров, включая AndroidX Media3/ExoPlayer, поддерживают HLS (ExoPlayer легко воспроизводит плейлист через `MediaItem.fromUri(hlsUrl)`). При необходимости низкой задержки можно применить **Low-Latency HLS**, но стандартный HLS (~8–12 сек. задержки) уже позволяет реализовать «DVR»-функции (запись эфира за прошлые часы) и удобный seek без пропуска аудио.

## Серверная архитектура

На сервере запускается фоновый процесс, который цепляется к входному HLS-стриму (https://sk.cri.cn/905.m3u8) через **FFmpeg**. Например, команда `ffmpeg -re -i <URL> -c copy -f hls -hls_time 3 -hls_list_size 3600` будет читать исходный поток в реальном времени и писать новый HLS-плейлист с трёхсекундными сегментами. Одновременно FFmpeg обеспечивает стабильную буферизацию и хранит последние ~3 часа аудио (3600 сегментов) в файловой системе. Плюс HLS-ноды легко масштабируются через стандартные HTTP/CDN.

Каждый сегмент (т. н. «3‑секундный файл») затем обрабатывается модулем речевого распознавания **sherpa-onnx** (модель SenseVoice). Sherpa-onnx по аудио-файлу выдаёт транскрибированный текст и **таймкоды** начала каждого символа. Например, для файла `zh.wav` декодер возвращает что-то вроде 
```json
{"text":"开放时间早上9点至下午5点。","tokens":["开","放","时","间","早","上","9","点","至","下","午","5","点","。"],"timestamps":[0.72,0.96,1.26,1.44,1.92,2.10,2.58,2.82,3.30,3.90,4.20,4.56,4.74,5.46]}
``` 
где каждый «token» – иероглиф (или цифра/знак), а `timestamps` – время в секундах от начала сегмента. 

Далее распознанный текст (последовательность иероглифов) передаётся в токенизатор **go-ego/gse**, который разбивает китайскую строку на слова (Chinese word segmentation). Например, «开放时间早上9点至下午5点。」 разобьётся на слова «开放», «时间», «早上», «9点», «至», «下午», «5点», «。」. Для каждого слова ищется пиньинь и перевод в словаре **CC-CEDICT**. Можно использовать готовый парсер CEDICT для Go (например, `go-cc-cedict`), который по упрощённой форме слова извлекает его пиньинь и англ.определения. 

Все эти данные (полный текст сегмента, списки слов, их тайминги, пиньинь и переводы) упаковываются в один JSON-файл, название которого отвечает порядковому номеру или временному метке сегмента. Файлы сохраняются в каталоге `~/tmp/china_radio_international` с TTL ~3 часа: старые автоматически удаляются (скриптом или флагом Docker), чтобы не засорять хранилище. Docker-образ сервера делается многоступенчатым: первый слой содержит Linux и все зависимости (FFmpeg, модели sherpa-onnx, гошные библиотеки gse и парсер CEDICT и т.п.), второй – только скомпилированный Go-бинарь. Это позволяет быстро перестраивать образ при изменении кода, не переустанавливая заново тяжёлые модели и пакеты.

## API и форматы данных

**Аудиопоток:** сервер отдаёт клиенту HLS-плейлист и сегменты по HTTP (например, статически раздавая файлы из Docker-контейнера или через Nginx). Клиент при нажатии «Play» просто начинает читать указанный URL (`/china.m3u8`).

**Субтитры:** для синхронизации слов на клиенте можно сделать REST-API. Например, `/api/subtitles?time={t}` или `/api/segment/{n}` возвращает JSON с информацией о словах текущего временного интервала. Формат JSON может быть таким:
```json
{
  "segment": 1234,
  "start_time": 3700.0,
  "duration": 3.0,
  "words": [
    {"text":"开放","start":0.72,"end":1.11,"pinyin":"kāifàng","translation":"открывать"},
    {"text":"时间","start":1.26,"end":1.68,"pinyin":"shíjiān","translation":"время"},
    ...
  ]
}
```
Здесь поле `start` – время начала слова внутри сегмента, `end` – приблизительно конец слова (можно взять время последнего символа плюс небольшой запас), `pinyin` и `translation` – из словаря CEDICT. Сервер должен уметь по времени запрашивать соответствующий сегмент и выдавать слова, что позволяет клиенту подсветить нужное слово. При нажатии на слово клиент может вызвать API `/api/wordinfo?segment=1234&word_index=5`, если нужно дополнительное инфо. JSON-файлы могут храниться на сервере несколько часов и пересоздаваться по мере поступления новых сегментов (TTL контролируется скриптом).

## Клиент на Android

Для воспроизведения аудио подойдёт AndroidX Media3 (ранее ExoPlayer). Он «из коробки» умеет играть HLS-потоки. В приложении создаётся `ExoPlayer`, указывается `MediaItem.fromUri(hlsUrl)` и вызывается `player.prepare()`. Интерфейс – простой, с кнопкой Play/Pause. Когда поток играет, приложение отслеживает текущее время `player.getCurrentPosition()` и по нему подсвечивает слово. Подсветка реализуется путём сопоставления таймкода плеера со списком слов из последнего полученного JSON. Текущее слово подсвечивается (например, меняем цвет текста).

При нажатии на слово показывается всплывающее окно с деталями: слово, пиньинь, перевод (из заранее полученного JSON). Кнопка «произнести» просто делает `player.seekTo(startTimeOfWord)` и проигрывает этот кусок; по тайм-коду слова (или его окончанию) можно программно приостановить плеер после завершения произнесения. Кнопка «+» дописывает слово в файл `Downloads/china_international_radio.txt`. Если нажать на неактивное (ещё не подсвеченное) слово, приложение ставит плеер на паузу, подчёркивает его и делает `player.seekTo` в начало этого слова. При возобновлении проигрывания звук начнётся именно с этого слова.

Также можно добавить настройку «Показывать ли пиньинь»: просто переключать видимость надстрочного текста. UI удобнее сделать на Compose или стандартных виджетах, но технически главное – иметь связь между аудио-временем и текстом. Media3/ExoPlayer поддерживает корректную буферизацию и pause/resume без потери sync, так что подсветка всегда будет в такт.

## Тестирование

1. **Надёжность и устойчивость:** проверить длительное (>3 часа) воспроизведение. Симулировать потерю пакетов и сбои сети (например, через Network Link Conditioner) – плеер должен корректно восстанавливать буфер из HLS-сегментов. Убедиться, что повторный `player.prepare()` или повторное нажатие Play после паузы не сдвигает синхронизацию текста.  

2. **Задержка:** измерить задержку «реального эфира» относительно оригинального потока. HLS даёт буфер ≈6–12 сек (можно сократить размер сегмента или использовать LL-HLS для уменьшения), но важно, чтобы она была стабильной. Проверить, что при просадках сети плеер не слишком часто перезапускает плейлист (т.е. буфер никуда не «убегает»). 

3. **Перемотка (seek):** многократно тестировать переход вперед/назад. При запросе предыдущих сегментов (например, час назад) убедиться, что JSON-подсказки есть на сервере (не удалены старые) и они верно применяются к аудио. После `seekTo` проверять, чтобы подсветка синхронизировалась уже с нового места. 

4. **Плавность звука:** проверить, нет ли шипения или рывков при смене сегментов в HLS, особенно после паузы/возобновления. Если возникают артефакты, можно увеличить буфер сегментов или использовать более мелкие сегменты и «запас» в плейлисте. 

5. **Корректность распознавания:** сравнить вывод sherpa-onnx (иерархия слов и символов) с ожидаемым текстом радио. При недочетах модели скорректировать postprocessing (например, регулировать `language` или `context` в Sherpa). 

Таким образом, описанная архитектура опирается на проверенные технологии (HLS/HTTP, FFmpeg, Sherpa-onnx, гошные библиотеки gse и CEDICT) и обеспечивает бесперебойную трансляцию с поддержкой субтитров и интерактивной перемотки. Это минимизирует риск «промаха» и позволит слушателям учить язык на китайском радио с комфортом.  

**Источники:** документация и статьи по HLS и потоковому аудио, примеры ExoPlayer на Media3, официальные примеры Sherpa-onnx и описания библиотек gse и CC-CEDICT.
