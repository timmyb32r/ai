# Проектирование надежной инфраструктуры для интерактивной радиотрансляции: от потокового протокола до клиент-серверной архитектуры на Go и Android

Данный исследовательский отчет представляет собой всесторонний анализ и проектное решение для создания клиент-серверной инфраструктуры, предназначенной для потоковой передачи китайского разговорного радио с богатым набором интерактивных функций. Целью является проектирование надежной, масштабируемой и простой в развертывании системы, состоящей из серверной части на языке Go и мобильного клиента для платформы Android. Основное внимание уделяется решению ключевой проблемы — выбору стандартизированного потокового протокола, способного доставлять аудиопоток вместе с синхронизированной пользовательской метаинформацией (иероглифами, пиньинем, переводами и временные метки) без рассинхронизации. Отчет детально рассматривает каждую составляющую системы: от выбора протокола и серверной логики до реализации интерактивного клиента и оптимизации инфраструктуры на базе Docker.

## Выбор Стандартизированного Протокола для Доставки Аудио и Метаданных

Выбор правильного протокола потоковой передачи является фундаментальным элементом успеха всего проекта. Пользователь столкнулся с трудностями при реализации собственного протокола, что привело к проблемам с буферизацией и рассинхронизацией, подчеркивая критическую важность использования проверенной, де-факто стандартной технологии [[1](https://stackoverflow.com/questions/30691684/whats-the-reason-of-using-circular-buffer-in-ios-audio-calling-app), [30](https://stackoverflow.com/questions/76895315/while-streaming-audio-chunk-by-chunk-it-has-pause-interval-between-chunks)]. Задача состоит в том, чтобы найти протокол, который не только обеспечивает стабильную доставку аудиопотока, но и предоставляет механизм для встраивания произвольных пользовательских метаданных с гарантией их строгой синхронизации с аудиоконтентом. Анализ показывает, что HTTP Live Streaming (HLS) является наиболее подходящим кандидатом для решения этой задачи.

HLS, разработанный Apple, является одним из самых распространенных и совместимых протоколов потоковой передачи в мире, поддерживаемого практически всеми устройствами, включая ПК, Android, iOS, SmartTV и OTT-приставки [[3](https://ossrs.net/lts/en-us/docs/v7/doc/hls)]. Эта широкая поддержка делает его идеальным выбором для кросс-платформенного приложения, где необходимо минимизировать сложность интеграции и обеспечить максимальную совместимость со сторонними устройствами. Для Android-разработчиков это преимущество особенно актуально, поскольку современный медиаплеер AndroidX Media3 имеет встроенную и хорошо документированную поддержку HLS, включая работу с live-потоками [[2](https://stackoverflow.com/questions/11019564/hls-http-live-streaming-on-android-3-0-and-seeking), [17](https://blog.csdn.net/ooo22/article/details/153496036), [26](https://blog.gitcode.com/fd1478c2fc0648208af3c244b86924bc.html)].

Ключевое требование проекта — возможность добавления пользовательской метаинформации — является одной из сильных сторон HLS. В отличие от некоторых других протоколов, HLS позволяет встраивать так называемые in-band timed metadata непосредственно в потоковые сегменты [[24](https://www.w3.org/TR/media-timed-events/)]. Это достигается путем добавления специальных метаданных, таких как ID3-теги, в каждый TS-сегмент аудиопотока. Эти теги могут содержать любые данные, которые затем декодируются клиентским плеером в точном соответствии с временной меткой, вставленной в поток [[21](https://stackoverflow.com/questions/54862783/dynamically-inject-id3-in-ffmpeg-live-stream)]. Поскольку метаданные и аудиоданные находятся в одном и том же сетевом пакете и имеют общую временную ось, обеспечивается их абсолютная синхронизация. Это полностью решает главную проблему пользователя о рассинхронизации. Когда клиентское приложение достигает момента времени, соответствующего началу слова, плеер может получить от сервера не только следующий фрагмент аудио, но и связанные с этим моментом данные (иероглиф, пиньинь, перевод), что позволяет немедленно обновить интерфейс, например, подсветить нужное слово [[23](https://stackoverflow.com/questions/29772013/http-live-streaming-how-to-listen-for-timed-metadata-embedded-as-id3-tags-using), [24](https://www.w3.org/TR/media-timed-events/)].

Поддержка ID3-тегов в HLS-потоках является частью спецификации и широко используется для доставки дополнительной информации, такой как название трека и исполнитель [[24](https://www.w3.org/TR/media-timed-events/)]. Хотя исторически поддержка этого механизма могла быть неоднородной в зависимости от платформы и версии плеера (например, на iOS существовали ограничения с использованием JavaScript API [[23](https://stackoverflow.com/questions/29772013/http-live-streaming-how-to-listen-for-timed-metadata-embedded-as-id3-tags-using)]), современные медиаплееры, такие как ExoPlayer из AndroidX Media3, демонстрируют хорошую способность обрабатывать такие метаданные [[26](https://blog.gitcode.com/fd1478c2fc0648208af3c244b86924bc.html)]. Это делает HLS с ID3-тегами жизнеспособным и предпочтительным решением для данной задачи. AWS MediaLive, мощная сервисная платформа для стриминга, также подтверждает поддержку встраивания ID3-метаданных для вывода HLS, предоставляя рекомендации по частоте вставки тегов (например, половину длины сегмента), что говорит о зрелости этого подхода [[22](https://docs.aws.amazon.com/medialive/latest/ug/insert-timed-metadata.html)].

В качестве альтернативы можно рассмотреть Dynamic Adaptive Streaming over HTTP (DASH). Этот протокол также имеет встроенные механизмы для доставки timed metadata и, теоретически, более строгие спецификации в этом отношении [[21](https://stackoverflow.com/questions/54862783/dynamically-inject-id3-in-ffmpeg-live-stream)]. Однако его практическое применение на Android исторически было менее стабильным, особенно для live-потоков, по сравнению с HLS [[16](https://blog.gitcode.com/a4bf69e3aca59441778646ff90425f4f.html)]. Кроме того, более высокий порог входа и меньшая распространенность делают HLS более безопасным и практичным выбором для проекта, ориентированного на быструю разработку и использование проверенных технологий.

Необходимо категорически исключить из рассмотрения самописные протоколы на основе TCP или UDP. Как отметил пользователь, этот путь привел к значительным трудностям. Создание надежного потокового протокола с управлением буферизацией, перебалансировкой, обработкой ошибок и гарантией синхронизации требует глубоких экспертиз в области сетевых технологий и является чрезвычайно сложной задачей [[1](https://stackoverflow.com/questions/30691684/whats-the-reason-of-using-circular-buffer-in-ios-audio-calling-app)]. Это создает высокий риск "выстрела себе в ногу", что прямо противоречит цели проекта — построить систему на минимальных рисках [[30](https://stackoverflow.com/questions/76895315/while-streaming-audio-chunk-by-chunk-it-has-pause-interval-between-chunks)]. Другой вариант, out-of-band метаданные, когда данные передаются через отдельный HTTP-запрос для каждой метки, также не подходит. Это приведет к огромной нагрузке на сеть и, что еще более важно, не позволит гарантировать синхронизацию, так как задержки на разных запросах будут различаться [[16](https://blog.gitcode.com/a4bf69e3aca59441778646ff90425f4f.html)].

Таким образом, на основании анализа совместимости, поддержки, возможности встраивания метаданных и степени зрелости экосистемы, **HTTP Live Streaming (HLS) с встраиванием пользовательских данных в виде ID3-тегов** является оптимальным и единственным разумным выбором для протокола между клиентом и сервером. Этот выбор обеспечивает все необходимые функции для реализации проекта, минимизируя риски и сложность.

## Архитектура Серверной Части на Go с Обработкой Речи

Серверная часть приложения, написанная на языке Go, играет роль центрального "умного" шлюза, который преобразует сырой аудиопоток из интернет-радио в структурированные, богатые семантической информацией данные, доступные для потребления клиентом. Архитектура сервера должна быть модульной, надежной и эффективной, чтобы справиться с задачами получения, обработки и трансляции данных в реальном времени. Пользователь уже определил ключевые технологии, и анализ подтверждает их адекватность и зрелость для выполнения поставленных задач.

Первым шагом является получение аудиопотока с исходного URL `https://sk.cri.cn/905.m3u8`. Этот URL указывает на M3U8-манифест, который является стандартом для потоковой передачи HTTP Live Streaming (HLS) [[3](https://ossrs.net/lts/en-us/docs/v7/doc/hls)]. Сервер должен периодически запрашивать этот манифест для отслеживания изменений и получения списка доступных TS-сегментов. Для загрузки этих 3-секундных сегментов можно использовать утилиту `ffmpeg`, которая является отраслевым стандартом для работы с мультимедийными потоками [[30](https://stackoverflow.com/questions/76895315/while-streaming-audio-chunk-by-chunk-it-has-pause-interval-between-chunks)]. Однако для обеспечения максимальной производительности и контроля лучше использовать Go-библиотеки, которые парсят M3U8 и последовательно загружают сегменты, передавая их в дальнейшую обработку без необходимости записывать их на диск. Это снижает задержку и нагрузку на файловую систему.

Центральным компонентом сервера является модуль распознавания речи. Для этой цели пользователь выбрал `sherpa-onnx` с моделью `SenseVoice` [[34](https://www.modelscope.cn/models/xiaowangge/sherpa-onnx-sense-voice-small), [36](https://www.bilibili.com/video/BV1Um421V75A/)]. Этот выбор является современным и эффективным. Sherpa-ONNX — это мощный и гибкий фреймворк для распознавания речи, работающий с моделями в формате ONNX (Open Neural Network Exchange), что обеспечивает кросс-платформенную совместимость и высокую производительность [[35](https://openvela.csdn.net/694c92935b9f5f31781a6fde.html)]. Модель SenseVoice специально разработана для задач распознавания речи с несколькими языками, что идеально соответствует требованиям проекта [[34](https://www.modelscope.cn/models/xiaowangge/sherpa-onnx-sense-voice-small)]. Критически важным является использование `sherpa-onnx` в режиме потоковой обработки (streaming mode), то есть онлайн-режиме. В отличие от офлайн-режима, который требует загрузки всего аудиофайла перед началом распознавания, онлайн-режим обрабатывает аудиопоток по мере его поступления, что обеспечивает минимальную задержку [[20](https://www.cnblogs.com/abyte/p/18550239)]. Это позволяет серверу начать генерацию текстовых выходов и временных меток для иероглифов практически одновременно с получением аудиосегмента. Go-библиотека `sherpa-onnx` активно развивается и поддерживает все необходимые функции для этой задачи, включая управление параметрами потока через `SetOption/GetOption` и официальную поддержку модели SenseVoice [[18](https://pub.dev/packages/sherpa_onnx_android_x86_64/changelog), [19](https://pypi.org/project/sherpa-onnx/1.10.46/)]. Существование демонстрационного приложения для Android также свидетельствует о зрелости и производительности всей экосистемы [[48](https://www.linkedin.com/pulse/vllm-ignites-kv-cache-war-sanchit-monga-tt3xc)].

Полученный текстовый выход от `sherpa-onnx` содержит не только сами иероглифы, но и временные метки, указывающие на начало и конец каждого распознанного фрагмента. Этот текст передается на следующий этап — токенизацию. Здесь используется библиотека `github.com/go-ego/gse` [[38](https://blog.csdn.net/gitblog_01100/article/details/141836725)]. GSE — это высокопроизводительная библиотека для разбиения текста на слова (токенизации) на разных языках, включая китайский [[40](https://bbs.itying.com/topic/676d38445b798701ddf8014e)]. Она поддерживает различные режимы, такие как обычный, поисковый и полный, что позволяет точно нарезать текст на отдельные слова. Полученные слова являются ключевыми единицами для последующей интерактивной обработки.

Далее, для каждого слова, полученного от `gse`, выполняется запрос к словарю CC-CEDICT. CC-CEDICT — это популярный открытый словарь китайско-английского перевода. Для обеспечения быстрого доступа его содержимое (пары "иероглиф -> пиньинь и перевод") следует загрузить в память при старте сервера, например, в виде хеш-таблицы (map в Go). Это позволит выполнять поиск для каждого слова за время $O(1)$, минимизируя задержки в общей цепочке обработки. В результате для каждого слова сервер будет иметь полный набор информации: само слово, его пиньинь и перевод.

Финальный и самый ответственный этап — упаковка сгенерированных метаданных и их доставка клиенту. Как было определено в предыдущем разделе, наиболее надежным способом является встраивание метаданных в аудиопоток в виде ID3-тегов. Для этого сгенерированные для каждого сегмента метаданные (например, JSON-объект, содержащий список слов с их временными метками, пиньинем и переводом) помещаются в ID3-тег и добавляются в соответствующий аудиосегмент. После этого сегмент помещается в директорию, доступную для веб-сервера (например, Nginx), который будет обслуживать итоговый HLS-поток. Такой подход обеспечивает, что клиент получает аудио и всю связанную с ним информацию в одном потоке, гарантируя их синхронизацию.

Чтобы система была легковесной и легко развертываемой, весь серверный сервис должен быть упакован в Docker-контейнер. Для ускорения процесса разработки и пересборки рекомендуется использовать многоступенчатый Dockerfile. Первый слой (базовый образ) будет содержать установленные системные зависимости, скомпилированные бинарные файлы `ffmpeg` и `sherpa-onnx`, а также весомые файлы моделей ASR и словаря CC-CEDICT. Этот слой меняется редко, поэтому он быстро кэшируется Docker. Второй слой будет содержать только скомпилированный бинарный файл Go-сервера. Этот слой будет пересобираться каждый раз при изменении исходного кода приложения, что значительно сокращает время сборки всего образа [[41](https://stackoverflow.com/questions/79486791/how-to-optimize-docker-multi-stage-build-for-faster-node-modules-copy), [42](https://dev.to/ruanbekker/why-you-should-use-multi-stage-docker-builds-571e)].

Наконец, необходимо реализовать механизм управления временными данными. Сгенерированные метаданные должны храниться во временном каталоге с TTL (время жизни) в 3 часа. Это полностью соответствует предназначению временных файлов, создаваемых для краткосрочного хранения данных [[7](https://superuser.com/questions/601414/how-long-do-temporary-files-made-with-mktemp-last)]. Для автоматического удаления старых файлов идеально подходит механизм Time-To-Live (TTL), который широко используется для управления размером баз данных и очистки устаревших данных [[5](https://www.linkedin.com/pulse/keeping-databases-lean-ttl-practical-insights-leela-kumili-qc2hc)]. Реализовать это можно с помощью фоновой горутины в Go-сервере, которая периодически (например, каждые несколько минут) сканирует директорию, проверяет время модификации файлов и удаляет те, чей срок жизни истек. Также важно предусмотреть корректную очистку при завершении работы сервера, возможно, с помощью сигнала прерывания или вызова `defer` [[4](https://stackoverflow.com/questions/33227063/best-way-to-clear-up-temporary-files)].

## Разработка Клиентского Приложения для Android с Использованием Media3

Разработка клиентской части для Android является критически важной, поскольку именно здесь пользователь взаимодействует с продуктом. Для достижения требуемого уровня стабильности, интерактивности и плавности воспроизведения единственно верным выбором является использование библиотеки AndroidX Media3, которая представляет собой развитие и современный стандарт для ExoPlayer [[12](https://blog.csdn.net/qq_35864421/article/details/115345091), [27](https://android-docs.cn/media/media3/exoplayer/migration-guide)]. Media3 предоставляет мощный, гибкий и официально поддерживаемый Google фреймворк для медиавоспроизведения, который полностью покрывает все требования проекта.

Основой клиента будет `ExoPlayer` из библиотеки Media3. Этот медиаплеер является заменой устаревшей `MediaPlayer` API и предлагает значительно больше возможностей, включая поддержку различных форматов, HLS и DASH, а также тонкую настройку поведения воспроизведения [[28](https://blog.csdn.net/yuantian_shenhai/article/details/127870700)]. Media3 выступает как высокоуровневый API поверх самого ExoPlayer, предлагая стабильные поверхности API и упрощая многие сложные операции, такие как управление медиасессиями, интеграция с системными панелями управления и голосовыми помощниками [[14](https://juejin.cn/post/7387016941552418870)]. Для данного проекта ключевыми преимуществами Media3 являются:
*   **Стабильное воспроизведение HLS:** Media3 отлично справляется с воспроизведением HLS-потоков, включая live-трансляции. Он корректно обрабатывает M3U8-манифесты, скачивает сегменты и обеспечивает плавное воспроизведение даже при колебаниях сетевой скорости [[43](https://help.aliyun.com/zh/linkvisual/methods-of-the-exoplayer-hls-player)].
*   **Управление буферизацией:** Media3 предоставляет API для настройки размера буфера (`PREFERRED_BUFFER_DURATION`, `PREFERRED_REBUFFER_DURATION`), что позволяет оптимизировать буфер под конкретные условия и избежать прерываний воспроизведения [[47](https://stackoverflow.com/questions/39512484/recommended-buffer-and-rebuffer-duration-of-data-for-exoplayer-instance-and-the)]. Это критично для обеспечения постоянного звука без пауз и шипения.
*   **Поддержка перемотки:** Media3 позволяет программно перематывать поток с помощью метода `seekTo()`, что является основой для реализации интерактивных функций, таких как перемотка на ранее прослушанное слово [[8](https://stackoverflow.com/questions/52329974/android-exoplayer-forward-rewind-programmatically)]. Важно понимать, что перемотка в HLS-потоке ограничена доступным "окном" на сервере, но этого достаточно для реализации функционала по перемотке на последние 3 минуты [[32](https://stackoverflow.com/questions/69538881/simpleexoplayer-in-android-not-allowing-to-seek-previous-part-of-video)].
*   **Работа с метаданными:** Media3 имеет встроенную поддержку обработки in-band timed metadata, в частности ID3-тегов. Это ключевая функция, которая позволит клиентскому приложению получать метаданные (текст, пиньинь, перевод) в нужный момент времени, синхронно с аудио [[24](https://www.w3.org/TR/media-timed-events/), [26](https://blog.gitcode.com/fd1478c2fc0648208af3c244b86924bc.html)]. Приложение сможет подписаться на события получения метаданных и обновлять интерфейс в реальном времени.

Логика клиента будет строиться вокруг `ExoPlayer` и его состояния. Главный экран приложения будет содержать `TextView` для отображения иероглифов, который будет динамически обновляться для подсветки текущего слова. Также на экране будут кнопки управления воспроизведением (Play/Pause) и переключатель для отображения/скрытия пиньиня.

Процесс взаимодействия пользователя с приложением будет следующим:
1.  **Воспроизведение и подсветка:** После начала воспроизведения, приложение будет периодически (например, каждые 100-200 миллисекунд) запрашивать у `ExoPlayer` текущее время воспроизведения (`player.getCurrentPosition()`). Это значение сравнивается с временными метками, полученными из ID3-тегов. Когда текущее время попадает в интервал времени одного из слов, соответствующий `TextView` в списке слов подсвечивается. При перемотке (`seekTo`) этот процесс повторяется, и подсветка немедленно смещается к новому слову.
2.  **Интерактивный словарь:** При нажатии на любой слово в интерфейсе, происходит следующее:
    *   Воспроизведение плеера приостанавливается (`player.setPlayWhenReady(false)`).
    *   Приложение запрашивает полную информацию об этом слове (пиньинь, перевод) из кэша метаданных.
    *   Открывается диалоговое окно (`DialogFragment`), отображающее эту информацию. В окне будут кнопки "Озвучить" и "+".
    *   Кнопка "Озвучить" будет вызывать метод, который извлекает из аудиопотока и воспроизводит только этот конкретный слово (что может потребовать дополнительной обработки с помощью `ExoPlayer` для извлечения сегмента) [[25](https://stackoverflow.com/questions/54842302/how-to-extract-a-section-of-audio-using-exoplayer)].
    *   Кнопка "+" добавит слово и перевод в конец файла `Downloads/china_international_radio.txt`.
3.  **Перемотка на слово:** Если пользователь нажимает на слово, которое не является текущим, приложение должно перемотать воспроизведение на начало этого слова. Это реализуется путем вызова `player.seekTo(windowIndex, wordStartTime)`. После перемотки, если плеер был в состоянии воспроизведения, его нужно снова запустить.

Для управления временными данными на стороне клиента (хранение последних 3 минут) можно использовать кольцевой буфер [[1](https://stackoverflow.com/questions/30691684/whats-the-reason-of-using-circular-buffer-in-ios-audio-calling-app)]. Буфер будет хранить в памяти последние несколько секунд аудиосегментов и соответствующие им блоки метаданных. Это позволит обеспечить быстрый доступ к данным для перемотки и отображения, не требуя постоянных обращений к сети.

В целом, AndroidX Media3 предоставляет все необходимые инструменты для реализации сложной интерактивной логики. Однако ее эффективность напрямую зависит от качества и своевременности данных, поставляемых сервером. Именно поэтому архитектурный выбор серверной части и протокола связи становится решающим фактором для успешной реализации проекта.

## Управление Временными Данными и Оптимизация Docker-Контейнера

Эффективное управление ресурсами и быстрое время сборки являются ключевыми факторами для успешной разработки и поддержки проекта. В данном случае эти два аспекта — управление временными файлами на сервере и оптимизация Docker-образа — тесно связаны и требуют продуманного подхода.

На серверной стороне, написанной на Go, основной объем данных, которые необходимо управлять, — это сгенерированные метаданные. Они создаются для каждого аудиосегмента и должны храниться во временном каталоге с ограниченным сроком жизни. Это полностью соответствует концепции временных файлов, предназначенных для краткосрочного хранения данных, которые обычно удаляются после завершения работы программы или по истечении определенного времени [[6](https://docs.python.org/3/library/tempfile.html), [7](https://superuser.com/questions/601414/how-long-do-temporary-files-made-with-mktemp-last)]. Использование механизма Time-To-Live (TTL) для автоматической очистки является наиболее надежным и эффективным подходом. TTL позволяет автоматизировать процесс удаления устаревших данных, предотвращая бесконтрольный рост дискового пространства и обеспечивая, что на сервере всегда будут доступны только самые свежие данные (в данном случае, метаданные за последние 3 часа) [[5](https://www.linkedin.com/pulse/keeping-databases-lean-ttl-practical-insights-leela-kumili-qc2hc)].

Реализация TTL-очистки на Go может быть выполнена с помощью фоновой горутины. Эта горутина будет работать постоянно в фоновом режиме и выполнять следующие действия:
1.  Перечислять файлы в указанной временной директории.
2.  Для каждого файла определять его время создания или модификации.
3.  Сравнивать это время с текущим временем. Если разница превышает установленный TTL (3 часа), файл помечается к удалению.
4.  Выполнять фактическое удаление помеченных файлов.

Такой подход обеспечивает пассивную и автономную очистку. Для предотвращения утечек ресурсов и обеспечения чистого завершения работы сервера, важно также реализовать обработку сигналов прерывания (например, SIGINT, SIGTERM). При получении такого сигнала, основной поток программы должен корректно завершить работу фоновой горутины и выполнить любые необходимые действия по очистке перед завершением. Использование `defer` для закрытия каналов или завершения ресурсов может помочь, хотя стоит отметить, что `defer` не сработает при использовании `os.Exit`, поэтому для нормального завершения следует использовать стандартные конструкции управления потоком [[4](https://stackoverflow.com/questions/33227063/best-way-to-clear-up-temporary-files)].

Второй важный аспект — это оптимизация Docker-образа. Пользователь прямо указал на необходимость двухслойной системы сборки для ускорения пересборки. Это стандартная практика в современной DevOps, направленная на минимизацию времени сборки Docker-образов путем эффективного использования кэша слоев Docker [[41](https://stackoverflow.com/questions/79486791/how-to-optimize-docker-multi-stage-build-for-faster-node-modules-copy), [42](https://dev.to/ruanbekker/why-you-should-use-multi-stage-docker-builds-571e)].

Многоступенчатый Dockerfile для данного проекта должен выглядеть примерно так:

```dockerfile
## Слой 1: Базовый образ с зависимостями
FROM golang:1.21 AS builder-base

# Установка системных зависимостей и утилит
RUN apt-get update && apt-get install -y ffmpeg
# ... установка других системных зависимостей ...

# Копирование моделей и словаря
COPY --chown=appuser:appuser models/ /app/models/
COPY --chown=appuser:appuser cedict/ /app/cedict/

## Слой 2: Сборка приложения
FROM builder-base AS builder-app
USER appuser

WORKDIR /app

# Копирование файлов go-приложения
COPY --chown=appuser:appuser go.mod go.sum ./
COPY --chown=appuser:appuser internal/ internal/
COPY --chown=appuser:appuser cmd/ cmd/

# Сборка бинарного файла приложения
RUN go build -o /usr/local/bin/radio-server cmd/server/main.go

## Слой 3: Финальный легковесный образ
FROM ubuntu:22.04 AS final

# Установка системных зависимостей для финального образа
RUN apt-get update && apt-get install -y ca-certificates && rm -rf /var/lib/apt/lists/*

# Создание пользователя
RUN adduser --disabled-password appuser
USER appuser

WORKDIR /home/appuser

# Копирование скомпилированного бинарного файла из слоя builder-app
COPY --from=builder-app /usr/local/bin/radio-server /usr/local/bin/

# Копирование временной директории и других файлов
RUN mkdir -p ~/tmp/china_radio_international
COPY --from=builder-base --chown=appuser:appuser /app/models/ ~/models/
COPY --from=builder-base --chown=appuser:appuser /app/cedict/ ~/cedict/

ENTRYPOINT ["/usr/local/bin/radio-server"]
```

Этот Dockerfile разделен на несколько этапов:
1.  **`builder-base`**: Этот слой содержит все тяжелые и редко изменяемые компоненты: базовый образ с системными библиотеками, утилиты (`ffmpeg`), модели для `sherpa-onnx` и файл словаря CC-CEDICT. Поскольку эти файлы редко меняются, Docker может кэшировать этот слой и переиспользовать его при каждом обновлении кода приложения, что значительно ускоряет сборку.
2.  **`builder-app`**: Этот слой собирает бинарный файл Go-сервера. Он копирует исходный код приложения. Этот слой будет пересобираться каждый раз, когда код приложения изменится.
3.  **`final`**: Этот слой создает финальный, максимально легковесный образ. Он копирует только необходимые исполняемые файлы и данные из предыдущих слоев, удаляя все лишние артефакты сборки. Это критически важно для снижения размера образа, повышения безопасности и ускорения его передачи по сети. Большой размер Docker-образов часто связан с наличием в них ненужных файлов, кэшей и крупных базовых ОС [[49](https://dev.to/latchudevops/why-is-my-ai-docker-image-so-big-a-deep-dive-with-dive-tool-to-find-the-bloat-l18), [50](https://www.linkedin.com/posts/vivek-tyagi-a7a210165_python-docker-machinelearning-activity-7459108859606061056-BGid)].

Применение этой многоступенчатой архитектуры с четким разделением зависимостей и приложения позволяет добиться очень быстрых итераций в процессе разработки, что полностью соответствует запросу пользователя. Серверная часть становится легковесной, управляемой и легко развертываемой, а клиентская часть, построенная на Media3, обеспечивает высокое качество пользовательского опыта.

## Комплексная Архитектура Системы и Практическая Реализация

На основе проведенного анализа можно сформулировать комплексную архитектуру системы "сервер-клиент" и предложить практический план действий для ее реализации. Предлагаемая архитектура основана на проверенных технологиях и стандартах, что минимизирует риски и обеспечивает надежность, масштабируемость и простоту развертывания.

**Комплексная архитектура системы**

Система состоит из четырех основных логических компонентов: источник аудиопотока, сервер обработки, веб-сервер для трансляции и клиентское приложение.

1.  **Источник аудиопотока:** Это интернет-радиостанция, предоставляющая поток по URL `https://sk.cri.cn/905.m3u8`. Этот URL является манифестом формата HLS, который состоит из серии коротких (в данном случае, 3-секундных) `.ts` файлов [[3](https://ossrs.net/lts/en-us/docs/v7/doc/hls)].

2.  **Сервер обработки (Go):** Это центральный узел системы, работающий в виде Docker-контейнера.
    *   **Модуль приема:** Парсит M3U8-манифест и последовательно загружает TS-сегменты. Использует `ffmpeg` или Go-библиотеки для этой задачи.
    *   **Модуль распознавания речи:** Передает каждый аудиосегмент в `sherpa-onnx` в режиме онлайн-обработки с моделью `SenseVoice`. Получает текстовый вывод с временными метками.
    *   **Модуль токенизации и словаря:** На текстовом выводе применяется `go-ego/gse` для разбиения на слова. Для каждого слова выполняется поиск в словаре CC-CEDICT (загруженном в память) для получения пиньиня и перевода.
    *   **Модуль упаковки и трансляции:** Сгенерированные метаданные (JSON-объект) встраиваются в аудиосегмент в виде ID3-тега. Готовый HLS-сегмент помещается в директорию, доступную для веб-сервера.
    *   **Модуль управления данными:** Фоновая горутина периодически сканирует директорию с сегментами и удаляет старые файлы, чей TTL (время жизни) истек (3 часа).

3.  **Веб-сервер:** Легковесный веб-сервер, такой как Nginx, который находится за пределами основного Go-сервера. Его задача — обслуживать итоговый HLS-поток (манифест и сегменты) по стандартному URL, например, `http://server-ip:8080/stream.m3u8`.

4.  **Клиентское приложение (Android):**
    *   **UI:** Главный экран с `TextView` для иероглифов, кнопками Play/Pause и переключателем для пиньиня.
    *   **Медиаплеер:** `ExoPlayer` из библиотеки AndroidX Media3, который подключается к URL веб-сервера (`http://server-ip:8080/stream.m3u8`).
    *   **Логика синхронизации:** `Player.Listener` отслеживает положение воспроизведения и события изменения таймлайна. По этим событиям приложение либо обновляет подсветку слова, либо запрашивает новые метаданные с сервера.
    *   **Интерактивность:** `OnClickListener` на словах обрабатывает нажатия, останавливает плеер и показывает диалог с информацией. Реализует логику перемотки (`seekTo`) при нажатии на неподсвеченное слово.

**Практический план реализации**

Рекомендуется следовать итерационному подходу, начиная с простого прототипа и постепенно добавляя сложность.

**Этап 1: Прототип обработки одного сегмента (Backend MVP)**
*   Создать простой Go-сервер, который принимает один TS-сегмент в качестве аргумента командной строки.
*   Реализовать его обработку с помощью `sherpa-onnx` (можно использовать командную строку утилиты `sherpa-onnx` для тестирования).
*   С помощью `ffmpeg` (командная строка) добавить в этот сегмент простой ID3-тег с тестовой информацией (например, `{ "word": "测试", "pinyin": "cè shì" }`).
*   Убедиться, что веб-сервер корректно обслуживает этот один сегмент и что он является частью валидного HLS-потока.

**Этап 2: Прототип клиента с получением метаданных (Frontend MVP)**
*   Создать простое Android-приложение с `ExoPlayer`, воспроизводящим HLS-поток с сервера.
*   Реализовать `Player.EventListener` для перехвата событий получения метаданных (ID3-тегов).
*   Выводить содержимое этих тегов в `Logcat` для проверки их получения и синхронизации с аудио.
*   Убедиться, что при перемотке в потоке метаданные также обрабатываются корректно.

**Этап 3: Реализация полноценной серверной логики**
*   Расширить Go-сервер для автоматического получения M3U8-манифеста и последовательной обработки всех входящих сегментов.
*   Интегрировать `go-ego/gse` для токенизации и чтение словаря CC-CEDICT для получения пиньиня и перевода.
*   Собрать и встроить в сегменты полные метаданные в формате JSON.
*   Реализовать механизм TTL-очистки для временных файлов.

**Этап 4: Разработка интерактивного UI и логики клиента**
*   Создать UI-компонент для отображения иероглифов и подсветки текущего слова.
